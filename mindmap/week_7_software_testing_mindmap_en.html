1. Purpose and limits of testing
▾
Four core purposes
  • Improve code quality
  • Spot code faults
  • Document behavior (living documentation)
  • Clarify requirements
Cannot guarantee full correctness ✗
  • Input space is effectively infinite
  • No finite test suite can exhaust it
  • “Guarantee fully correct” is a wrong testing goal
Tests at any lifecycle stage
  • Before — TDD/BDD drives design
  • During — add tests alongside code for coverage
  • After — regression tests after fixes
  • “All of the above” is often the correct exam answer
Test code volume
  • Test code can exceed production code size
  • UIs can be tested too
  • Tests need not always be code (e.g. checklists)
  • Testing cannot “always find every bug”
2. Four test activities
▾
Test Design
Design cases: pick inputs and expected outputs.
  • Requires solid domain and code knowledge
  • Not “only a little knowledge”
Test Automation
Encode test values in scripts for repeatable runs.
  • Useful but authoring can be tedious
  • That tradeoff statement is true
Test Execution
Run tests and collect results.
Test Evaluation
Analyze outcomes, pass/fail, and coverage.
“Test Implementation” ✗ not a standard type
Not one of the standard activity types—a common distractor.
Manual vs. automated
  • Frequent regressions → favor automation
  • Automate UI tests after the UI stabilizes
  • Automation ≠ always finding more bugs ✗
  • One-off checks → manual may suffice
Who writes tests
  • Developers have blind spots on their own code
  • Encourage devs, QA, and domain experts
  • “Only QA may write tests” ✗
  • Writing cases clarifies requirements
3. Black-box vs. White-box Testing
▾
Black-box Testing
  • External view; no internal knowledge required
  • Also called functional / specification-based
  • Cannot exhaust inputs—fundamental limit
  • Weaker at extra-logic faults
  • Weak for low-level issues like out-of-bounds
White-box Testing
  • Internal view; full code structure
  • Also called structural / code-based
  • Needs strong coding skill (not “easy” ✗)
  • Costly to maintain when code churns
  • Good for extra logic, dead code
Shared principles
  • ECP and BVA apply to both black- and white-box testing
  • They are not black-box–only techniques
4. PIE model — fault → failure chain
▾
P — Execution
Code containing the fault must run. If it never executes → no failure.
I — Infection
After execution the state must become wrong (infected). If state stays correct → no visible failure.
E — Propagation
The infected state must reach an observable output.
Exam trap: “infected state must NOT propagate” ✗—propagation is required.
Error (human mistake)
Wrong human decision leading to bad code.
Example: meant Math.log(5) but coded 5³.
Fault (defect in code)
The bug sitting in source—static defect.
Failure (observed wrong behavior)
What you see at runtime, e.g. 125 instead of ln(5).
5. Coverage metrics & techniques
▾
Standard coverage metrics
  • Statement coverage
  • Branch coverage
  • Method / function coverage
  • Condition coverage
Parameter coverage ✗ not standard
Not a standard metric—parameters are not an execution structure. Common distractor.
Equivalence Class Partitioning (ECP)
  • Partition inputs into equivalence classes
  • Inputs in one class behave the same
  • Pick one representative per class
  • Keywords: partition · representative
Boundary Value Analysis (BVA)
  • Focus on boundaries of classes
  • Test min, max, and boundary ±1
  • Keywords: boundary · ±1
  • Often paired with ECP
Decision table testing
Tabulate condition combinations and actions—good for complex rules.
Use case testing
Derive tests from user scenarios to validate behavior end-to-end.
Exam quick hits
  • Testing can prove full correctness ✗
  • “Test implementation” is a standard activity ✗
  • Parameter coverage is a standard metric ✗
  • Propagation in PIE is optional ✗
  • Automation always finds more bugs than manual testing ✗
  • Only QA may write test cases ✗
