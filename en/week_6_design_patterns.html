Week 6 · OO Design Patterns — Part 2

Design patterns — what are they?
Design pattern Reusable template Recurring problem GoF
Design patterns are reusable solution templates for recurring problems in software development. The Gang of Four (GoF) systematized 23 classic patterns in Design Patterns: Elements of Reusable Object-Oriented Software.
  • They capture structure and intent, not finished code you paste in.
  • Used well, they improve readability, flexibility, and maintainability.
  • A single class can participate in several patterns at once.
  • They give teams a shared vocabulary for talking about design.
Exam tip
“A class cannot participate in more than one design pattern” is false: a class can play multiple pattern roles at the same time.

Creational patterns — how objects are created
Creational Object creation Instantiation Flexibility
Creational patterns separate object creation from the code that uses those objects, so the rest of the system does not depend on concrete constructors. That improves flexibility and makes it easier to extend or swap implementations.
PatternCore idea
SingletonEnsure one instance of a class and provide a global access point.
Factory MethodDefine an interface for creating objects; let subclasses choose which class to instantiate.
Abstract FactoryProvide an interface for creating families of related objects without naming concrete classes.
BuilderConstruct a complex object step by step; the same construction process can yield different representations.
PrototypeCreate new objects by cloning an existing instance (e.g. clone()).
Structural patterns — how classes and objects are composed
Structural Composition Interface Composite
Structural patterns show how to build larger structures from classes or objects while keeping the design flexible and efficient. This week emphasizes the Composite pattern.
Composite lets a single object (leaf) and a group of objects (composite) be treated the same way by clients, forming a tree.
Component
Abstract type shared by all parts of the tree; declares operations every node supports.
Leaf
A leaf has no children; it implements the Component operations.
Composite
A composite holds child components and implements child-management operations.
Key distinction
The Component abstraction declares only operations shared by every node (e.g. render, resize).
Child-management methods (add / remove / getChild) live on the Composite class, not on the Component interface.
Typical scenario
A 3D scene graph: one mesh (leaf) and a group of meshes (composite) must support the same transforms (resize, rotate) — Composite lets clients apply operations uniformly to individuals and groups.
Behavioral patterns — collaboration and responsibility
Behavioral Interaction Responsibility Algorithm
Behavioral patterns describe how objects communicate and divide responsibility: encapsulating algorithms, coordinating collaborators, and changing behavior at runtime without brittle conditionals.
PatternCore ideaKeywords
ObserverWhen a subject’s state changes, notify all dependent observers automatically.Publish–subscribe, one-to-many notification
StrategyEncapsulate a family of algorithms and swap them at runtime.Runtime switching, algorithm family
IteratorTraverse a collection’s elements sequentially without exposing its internals.Traversal, sequential access
InterpreterDefine a grammar and represent rules as a hierarchy of expression classes.AST, grammar, syntax tree
VisitorAdd new operations over an object structure without changing element classes.Double dispatch, open operations
StateDelegate behavior that varies with internal state to state objects.State machine, transitions
Observer
Observer Subject Notify Decouple One-to-many
Observer decouples who holds state (subject) from who reacts to it (observers). When the subject changes, it notifies every registered observer.
  • Many observers can listen to the same subject (one-to-many).
  • The subject may change state at any time; observers receive updates after each change.
  • Central definition: separate the display of object state from the object itself — the subject holds state; observers react.
Contrast
Observer separates state from views of that state. It is not mainly about separating a view from its implementation — that line of thinking fits MVC or Strategy better.
Strategy
Strategy Algorithm family Runtime switch Encapsulation Interchangeable
Strategy defines a family of algorithms, encapsulates each one, and lets clients use them interchangeably at runtime without changing calling code.
  • Decouples using an algorithm from implementing it.
  • A context object holds a strategy reference and can reassign it when behavior should change.
  • Reduces sprawling if/else or switch chains over algorithm variants.
Typical scenario
A trading desk swaps execution algorithms during a session — each strategy is its own object; the context selects which one runs without rewriting core logic.
Interpreter
Interpreter Context AbstractExpression TerminalExpression NonTerminalExpression AST
Interpreter defines a language grammar and maps it to a hierarchy of expression classes. Evaluation walks an abstract syntax tree (AST) built from those expressions.
RoleResponsibility
AbstractExpressionDeclares interpret(Context) (or equivalent).
TerminalExpressionLeaf in the grammar; no sub-expressions; evaluates directly.
NonTerminalExpressionInterior node; holds child expressions and delegates interpretation recursively.
ContextHolds global information for interpretation; it is not a node in the AST.
Exam tip
The AST consists of expression objects, not Context instances.
Context is passed along during interpretation; it does not form the tree.

Common trap
“The component interface includes methods to manipulate children” is false: child-management methods belong on Composite, not on the shared Component abstraction.

Strategy vs. Visitor
Both can move behavior out of a monolithic class, but they solve different problems: Strategy swaps algorithms (whole behavior changes). Visitor adds new operations across a fixed object structure without changing each element’s class.
