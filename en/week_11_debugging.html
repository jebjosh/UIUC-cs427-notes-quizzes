Chapter 11 — Debugging

Failure, Fault and Debugging
FailureFaultDebugging
TermMeaning
FailureIncorrect behavior or output observable at runtime — the symptoms of a problem.
FaultA defect in the code or design — the underlying root cause that can lead to failures.
Debugging aims to find and fix faults. Testing aims to discover failures. The two have different purposes.
Exam tip
On “purpose of debugging,” choose fix faults — not “fix failures” or “find faults.”
Debugging Process
ReproduceFocusIsolate
Typical steps include reproduce, focus, isolate, and more. If you see “Which is NOT part of the debugging process,” read every option carefully — for example, None of the above means every listed step actually belongs to debugging.
  • Debugging and testing serve different purposes; they are not the same activity (a common False question).
Bug Report and Problem Tracker
BugzillaDuplicateHistory
TopicDescription

Problem historyMay include reproduction steps and other timeline details; a report can reference duplicate bug IDs.
Steps in Tracking / Reproducing
NeutralizeDisambiguateGeneralizeIsolate
StepRole (what to remember)
NeutralizeStrip out negative, irrelevant, or emotional wording; keep the technical facts (“remove negative information”).
DisambiguateRemove ambiguity from the problem description.
GeneralizeBroaden conditions while the failure still reproduces (e.g. it still crashes with several different fonts).
IsolateNarrow the scope and pin down necessary conditions (e.g. “ONLY on Windows XP”).
Exam tip
Isolate and generalization can be used together; they are not mutually exclusive (True/False items that claim they never appear together are often False).
Failing-Input Simplified: Binary Search & Delta Debugging
Binary SearchDelta Debugging
  • To find the failing line in a large input, binary search needs about log₂(n) splits (n = number of lines).
  • After binary search, you can still apply delta debugging to shrink the input further.
  • Delta debugging: try a few large chunks of change first, then move toward more numerous, smaller changes.
Fault Localization Policy
BacktrackingTracingCause Elimination
StrategyTypical use
BacktrackingFollow the error-propagation chain backward toward the fault. It usually does not mean “run instructions in reverse order.”
TracingUse breakpoints, assertions, print, single-stepping, etc. to observe execution.
Cause eliminationSystematically rule out candidate causes; requires program knowledge and experience with common mistakes.
Summary
Common tracing techniques include print debugging, breakpoints, assertions, and single-stepping. Exam questions may expect All of the above.
Assertions: Debug vs Test & Production Code
DebugTestProduction
Assertions in debugging vs. testing can differ in how and where you use them (many True/False questions say this is True).
Takeaway
Assertions generally should not be relied on as ordinary control-flow logic in production code (exams often expect No).
Design by Contract
PreconditionPostconditionInvariant
  • Postcondition: an assertion that should hold after the method finishes.
  • Invariant: in this course, often tested as “always true” over the relevant scope (many T/F answers are True).
  • Loop invariant: must hold before the loop, before each iteration, and when the loop exits (match the wording to the quiz options).
