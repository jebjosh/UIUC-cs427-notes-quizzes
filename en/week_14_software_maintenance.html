TypeCategoryPurposeCustomer request?Visible to user?

AdaptiveEnhancementAdapt to new environmentSometimesMaybe

Exam tip
Corrective and Preventive both fix problems — but Corrective reacts to discovered faults while Preventive is proactive. Adaptive responds to environment changes (new OS, new regulation). Perfective adds new features.
Reality of Software
Software always changes after delivery. Google's codebase: ~2 billion lines, 9 million source files, ~40,000 commits per workday. Maintenance is the longest and most expensive phase of the software lifecycle.
Key rule
All four maintenance types require testing — even Preventive (confirm nothing broke).
Size Metrics — Lines of Code (LOC)
LOC / SLOCCyclomatic Complexity
LOC (aka SLOC) is easy to measure and correlates with build/maintenance time. However it is only a valid metric when: same language, standard formatting, code already reviewed.
  • Language matters: Assembly ~3× longer than C; C ~3× longer than Java.
  • Counting blank lines and comments differently produces incomparable numbers.
  • Between two programs of equal functionality, fewer LOC is better.
Cyclomatic Complexity (McCabe, 1976)
Measures logical complexity at the method level. Upper bound on tests needed for complete branch coverage.
FormulaWhen to use
#decisionPoints + 1Exactly one exit point
#edges − #nodes + 2Multiple exit points (draw control flow graph)
#regions of flow graphGraphical alternative — count enclosed regions
  • Decision points: if, while, for, case, catch — one each.
  • Compound condition if (x > 0 && y > 0) counts as 2 decision points.
  • Cyclomatic complexity > 10: method is considered hard to test and fault-prone.
Example
processInterest(): 1 for + 1 if = 2 decision points → complexity = 3.
Uses
Identify refactoring candidates; focus code review and testing effort on high-complexity methods.

What is Refactoring? (Fowler et al., 1999)
Extract MethodExtract VariableIntroduce Parameter Object
Changing internal structure without changing external behaviour. Makes code easier to understand and cheaper to modify.
Key rule
Change DESIGN XOR FUNCTIONALITY — never both simultaneously. Refactoring must pass all existing tests. Changing behaviour requires new tests.
  • Key part of XP, software evolution, and making reusable software.
  • Refactor before changing behaviour, after changing behaviour, or to understand code.
  • One refactoring often leads to another (e.g., Introduce Parameter Object → Extract Method).
Extract Method
Group code that belongs together into a new method with an informative name. Increases reusability and readability.
  • Step 1: Create new method; name it by its purpose.
  • Step 2: Copy extracted code into new method.
  • Step 3: Replace original code with a call to the new method.
  • Step 4: Pass local variables used in extracted code as parameters.
  • Step 5: If one local variable is modified, return it. If more than one, cannot extract as-is.
Extract Variable
Replace a complex expression with a named final temporary variable that explains its purpose.
  • Declare a final temp variable set to the complex expression (or part of it).
  • Replace the expression with the temp variable.
Example
A complex boolean: platform.toUpperCase().indexOf("MAC") > -1 → final boolean isMacOs = ...
Introduce Parameter Object
When a group of parameters naturally go together and appear in many methods, create a new class for the group.
  • Step 1: Make a new class for the group (e.g., DateRange for start, end).
  • Step 2: Add a parameter of the new type to the method; keep old params temporarily.
  • Step 3: Update all callers to pass a new object.
  • Steps 4–5: Incrementally replace each old param with the new object's getters.
Follow-up
After Introduce Parameter Object, check if the condition logic should be moved as a method into the new class (Extract Method again).
